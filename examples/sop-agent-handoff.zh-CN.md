# 示例：SOP 与 Agent 交接

[English](sop-agent-handoff.md)

本示例使用虚构的汇总记录，展示一个重复社区动作如何从证据走到确认完成。自动化流程运行成功，不能直接证明公开动作已经完成。

## 场景

- 一个为期五天的学习活动已有 14 份有效投稿。
- 运营规则规定：如果第四天的有效投稿仍少于 20 份，准备一次提醒。
- 发布前，负责人必须批准准确文案和目标位置。
- 平台工具可以发布提醒，但不能修改角色、发送私信或结算奖励。

## SOP 合同

**操作**

准备、批准、发布并验证第四天的活动提醒。

**触发条件**

已验证的活动记录显示当前为第四天，且有效投稿少于 20 份。

**决策负责人**

社区运营者批准准确的提醒文案和目标位置。

**输入和唯一事实来源**

- 活动记录：当前天数、有效投稿数、目标位置和当前提醒状态；
- 已批准的活动规则：提醒阈值和最高发送频率；
- 字段缺失或过期时标记为 `data gap`，并停止准备。

**执行角色**

- 运营者：确认准确的提醒文案和目标位置。
- AI 或 Skill：验证触发条件并准备提醒交接包。
- 自动化流程：检查阈值并创建待审批事项。
- 平台工具：获批后发布一次，并返回执行凭据。

**执行步骤**

1. 读取活动记录，确认当前为第四天、已有 14 份有效投稿，并且此前没有发送提醒。
2. 准备一条提醒，将状态标记为 `pending approval`。
3. 记录负责人对准确文案和目标位置的批准。
4. 将获批的交接包发送给平台工具一次。
5. 回读目标位置，确认提醒标识和可见文案。
6. 将动作标记为 `verified`，把执行凭据写回活动记录，并纳入下一份报告。

## 交接包

```text
操作：第四天活动提醒
状态：approved
证据：第四天有 14 份有效投稿；此前没有提醒
目标：当前活动阵地
内容：负责人批准的提醒版本
允许的执行路径：已配置的平台工具
成功标准：目标位置只出现一条与批准版本一致的提醒
停止条件：目标变化、内容变化、缺少批准或发现此前已有提醒
验证方式：回读目标位置
写回位置：活动运营记录
```

## 正常结果

```text
prepared -> pending approval -> approved -> executed
-> verified -> recorded -> reviewed
```

只有回读目标位置并确认提醒可见后，日报才能写“提醒已发布”。

## 结果不明确

假设平台工具在提交后超时：

1. 将本次尝试标记为 `ambiguous`。
2. 保存目标位置、批准版本、尝试时间和平台工具响应。
3. 不要再次提交提醒。
4. 回读目标位置。
5. 如果提醒已经存在，验证并记录；如果不存在，把是否再次尝试的决定交还负责人。

这样可以避免一次超时造成重复公开提醒。
